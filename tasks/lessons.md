# Lessons

- 2026-04-30: Display preference changes saved from the standalone Settings window only updated that window's frontend store, so the already-running tray kept stale theme, used/left, reset timer, or menubar icon state until reload. Fix: publish `display-preferences:updated` over Tauri events from the shared display-settings action, let the tray update its preference store immediately, and reload persisted display preferences on panel focus as a missed-event fallback. Prevention: every settings value that affects visible state in multiple Tauri windows needs either a cross-window event or a backend replay path, not just persisted store writes.
- 2026-04-30: A provider can use the right vendor domain while still depending on undocumented internal/app usage endpoints, and docs that only say "API endpoint" overstate support maturity. Fix: reclassify Z.ai and Warp docs/source notes to distinguish legitimate hosts from undocumented subscription/quota or app GraphQL paths, keep both providers experimental, and verify UI copy/tests match the risk. Prevention: provider docs must name endpoint provenance explicitly: official public API, documented private/admin API, or reverse-engineered app/internal endpoint.
- 2026-04-28: Host capability and env allowlist changes can leave provider code with test-only success if plugin mocks expose APIs or env vars that the real Tauri host still blocks. Fix: add Cursor's explicit `sqliteWrite` capability, whitelist the new Copilot billing env vars in `host_api.rs`, and cover the Cursor manifest contract in a focused plugin test. Prevention: every provider change that adds a host API, env var, or write path must update the real host allowlist/capability manifest and include at least one test that reads the declared contract.
- 2026-04-28: Provider usage allowances and credit-pool totals can look like subscription tiers but are not authoritative plan identifiers. Fix: stop inferring Factory, MiniMax, and Perplexity plan labels from usage quantities, and only render plan names when the provider explicitly reports them. Prevention: provider showcase copy and plan badges must distinguish reported plan fields from derived usage buckets; do not invent subscription names from limits unless the provider documents that exact mapping and the code names the mapping source.
- 2026-04-24: Provider changes made in the standalone Settings window can leave the tray webview stale because each Tauri window owns its own frontend store. Fix: publish provider settings changes over a Tauri event, let the tray apply them immediately, and keep provider row selection on the visible tray-reveal path. Prevention: cross-window settings writes must update both persistence and the already-running target window state before relying on navigation events.
- 2026-04-03: Syncing the tray target on provider selection was still not enough because the Settings window stays hidden over the tray while it is open. Fix: intercept the Settings close request, reveal the selected provider in the tray, and hide Settings on the way out. Prevention: when a hidden settings surface controls the visible tray view, verify the close path, not just the selection callback.
- 2026-04-01: Settings provider selection in the dedicated window was only doing the hard tray handoff, which left no path for a live tray target sync without hiding Settings first. Fix: split the selection flow into a live `sync_panel_view` bridge plus an explicit `Open in tray` action, and keep row clicks from forcing the window close. Prevention: when one window edits a target shown in another hidden window, separate "update target state" from "show the target now" and test both paths independently.
- 2026-04-01: A Tauri updater endpoint pointed at GitHub's `releases/latest/download/latest.json` can still fail on a perfectly valid beta release, because GitHub's `latest` alias ignores prereleases even when the prerelease assets include `latest.json`. Fix: skip updater checks in local dev and semver-prerelease app versions until the release channel is stable or the endpoint strategy changes. Prevention: when wiring auto-updates to GitHub Releases, validate the exact release-channel semantics of the download alias before enabling updater checks for prerelease builds.
- 2026-04-01: Zed's local Windows credential target was readable, but the observed client token still returned `401` against the dashboard billing endpoints, so a keychain-first design did not automatically imply a working direct usage API. Fix: ship Zed as a local-telemetry provider backed by the external credential target plus `telemetry.log`, and document that dashboard spend parity is intentionally absent for now. Prevention: when reverse-engineering a desktop provider, validate the exact locally stored credential against the target API before committing the provider design to a remote usage path.
- 2026-03-28: Antigravity's backend already supported offline Cloud Code fallback from stored SQLite/OAuth data, but the settings copy still told users to "open Antigravity locally" as if the IDE had to stay open. Fix: align the provider summary/connect/status hints and docs with the actual no-IDE path, then add a settings regression for the offline guidance. Prevention: whenever a provider has a live-process fast path plus an offline credential path, keep the UI wording explicit about which part actually requires the app to be running.
- 2026-03-27: Cursor's current individual-plan usage payload can omit `planUsage.limit` while still sending `totalPercentUsed`, and it also includes zero-valued team-pool fields like `spendLimitUsage.pooledLimit: 0`; treating the mere presence of that numeric field as a team signal forced the UI into the `Total usage limit missing from API response.` error on valid personal accounts. Fix: classify Cursor team plans only when `planName`/`limitType` says team or the pooled limit is a positive number, and cover the exact live payload with a regression test. Prevention: when provider APIs ship optional numeric fields, distinguish `present` from `meaningful` in account-type heuristics and verify against a sanitized live payload before tightening validation.
- 2026-03-27: Landing a provider only in `plugins/<id>` is not enough because Tauri still executes the copied bundle under `src-tauri/resources/bundled_plugins`, so the desktop app can keep running a stale placeholder even while the source plugin/tests look correct. Fix: after provider/plugin manifest changes, rerun the bundle-copy step and verify the bundled plugin files match the source implementation before marking the rollout done. Prevention: every provider rollout slice that changes `plugins/<id>` must also audit or refresh `src-tauri/resources/bundled_plugins/<id>`.
- 2026-03-27: Leaving the GitHub issue link in the tray sidebar kept a configuration/support action in the quick-usage surface even after settings moved into a dedicated window. Fix: remove the tray help button and place `Report an issue` in `Settings > General`, with focused tests for both surfaces. Prevention: when an action belongs to app management rather than usage inspection, keep it in Settings instead of the tray nav.
- 2026-03-26: Hiding Windows child processes with `CREATE_NO_WINDOW` was not enough for the packaged Claude/Codex refresh path because invoking the Bun shim as `bunx.exe` could still surface a top-level console and steal focus from other apps. Fix: keep the hidden-process flags, but on Windows resolve/run the Bun runner as `bun.exe x ...` instead of `bunx.exe`, while caching the resolved runner list so refreshes stop re-launching the shim probe path. Prevention: when a Windows package runner is both a shim and a visible process in user reports, prefer the underlying real binary/subcommand pair over the shim executable and verify the packaged `.exe` path, not just Rust unit tests.
- 2026-03-24: A packaged Windows launch can still flash `bunx.exe` or other helper consoles even when the main Tauri app uses `windows_subsystem = "windows"`, because background child processes spawned from the Rust host API inherit normal console creation flags unless they are explicitly hidden. Fix: apply `CREATE_NO_WINDOW` to Windows-only background shell-outs in `src-tauri/src/plugin_engine/host_api.rs`, especially the `ccusage` runner detection/execution path used by Claude/Codex startup probes. Prevention: every Windows background subprocess in the packaged app should go through one helper that sets hidden-process flags instead of relying on the parent process to suppress consoles.
- 2026-03-24: Settings-to-tray provider switching could still look broken on Windows because the Settings window requested `show_panel_for_view` but stayed open, so the tray handoff remained hidden behind the still-focused Settings surface. Fix: hide the Settings window before invoking the tray provider reveal and cover the full handoff path in the settings-window tests. Prevention: for cross-window navigation, verify the visible window handoff, not just the command/event invocation.
- 2026-03-24: The first Windows beta release path can appear broken even after Tauri successfully emits the NSIS setup `.exe`, because the fork keeps updater signing enabled in `tauri.conf.json` and the local helper used to fail hard when `TAURI_SIGNING_PRIVATE_KEY` was missing. Fix: detect missing signing key in the local release helper and add `--no-sign` automatically for unsigned local beta builds, while keeping CI/tagged releases signed through secrets. Prevention: when this fork needs locally testable Windows installers before release secrets are wired in, make the local build path explicitly degrade to unsigned artifacts instead of treating missing updater-signing secrets as a build failure.
- 2026-03-23: A direct API provider can still stay broken on Windows if the plugin reads env vars that the host API does not whitelist and the settings copy still describes it as a placeholder. Fix: land the provider plugin, `provider-settings.ts`, and `src-tauri/src/plugin_engine/host_api.rs` allowlist update in the same slice, then verify both JS tests and the Rust env-allowlist test. Prevention: whenever a new provider depends on env vars or app-owned secrets, audit the host env allowlist and setup copy before calling the rollout done.
- 2026-03-23: OpenCode Go looked ready on paper but still failed on this Windows machine because the live CLI now writes `opencode` in `auth.json` and SQLite `providerID`, while the imported plugin still only accepted the older `opencode-go` identifiers. Fix: treat `opencode-go` and `opencode` as compatible local identifiers, then verify against the real `auth.json` and `opencode.db` before surfacing the provider. Prevention: when importing a local-history provider from upstream, verify the actual current on-disk identifiers on this machine instead of assuming the documented provider id is still the persisted schema.
- 2026-03-23: API-key providers can still ship misleading Windows setup UX if their settings metadata inherits the generic "sign in locally" copy path. Fix: align `provider-settings.ts`, provider docs, README status, and manifest support messages around the actual env-var setup path before surfacing the provider as experimental. Prevention: whenever a provider is auth-by-env rather than auth-by-session, audit the settings `connectHint` and status text in the same slice as the manifest rollout.

- 2026-03-22: Antigravity could still look IDE-bound on Windows even with valid local SQLite credentials because the stored Cloud Code bearer tokens had gone stale, the refresh token was the only live offline credential, and the plugin deferred refresh until after trying every stale token path. Fix: resolve Cloud Code auth in freshness order, treat cached/protobuf access tokens as disposable if they are expired or rejected, and promote refresh-token recovery ahead of the stale `apiKey` fallback. Prevention: for offline OAuth-backed providers, do not let expired or previously rejected bearer tokens outrank a known-good refresh path.
- 2026-03-22: Even after moving tray anchor math into Rust, the popup could still show slight bottom-edge drift on Windows because the final logical bounds were rounded into physical position and physical size separately, and the native path still exposed a non-atomic move/size update. Fix: convert the final logical tray bounds into one physical outer-bounds struct, preserve the bottom edge during DPI rounding, and apply the Windows update through one `SetWindowPos` call from the final size/position. Prevention: for bottom-docked Windows flyouts, treat physical-pixel rounding as part of the anchor contract and apply the final outer bounds in one native call instead of mixing logical position rounding with separately rounded height.
- 2026-03-22: Tray provider switching still looked like a whole-shell reload even after native anchor fixes because refresh/loading state cleared `pluginStates[id].data`, `use-app-plugin-views` could briefly expose empty nav/selection results during bootstrap transitions, and `use-panel` was measuring shell churn instead of settled intrinsic content. Fix: retain the last successful provider payload during refresh/error states, keep the last resolved nav/selected-provider snapshot for transient bootstrap gaps, and observe content/footer nodes with coalesced resize scheduling instead of the outer shell. Prevention: for tray UIs, do not let loading-state transitions destroy the last renderable view model or drive native window geometry directly from intermediate React layout states.
- 2026-03-22: The tray popup could still visibly drop/jump on Windows even with a stored bottom anchor because the frontend issued separate native `setSize` and `reposition_panel` calls; one intermediate frame always used the wrong fixed edge during height changes. Fix: move final tray bounds application into one backend-owned command that computes the top from the stored bottom anchor and applies the final size/position together from Rust. Prevention: for bottom-bound taskbar flyouts, do not split resize and reposition across independent frontend/native steps; compute final bounds once from the anchored edge and apply them in one backend pass.
- 2026-03-22: After fixing the wrong anchor, provider switches still felt unstable because the shell snapped directly to each view's natural height and the content swap happened as the same hard cut. Fix: add a stronger shared detail baseline, move large height changes through a short bounded tween, and soften the inner content transition without remounting the shell/sidebar. Prevention: once bottom anchoring is correct, treat shell-height variance and content-swap timing as separate UX problems; do not let every provider page dictate one-step popup height changes.
- 2026-03-22: Surfacing a hidden provider in `list_plugins` is not enough if the desktop bundle step still excludes that provider from `src-tauri/resources/bundled_plugins`; the UI can mark it loading while the backend has nothing to execute. Fix: keep provider surfacing and plugin bundling aligned, and verify the bundled plugin folder exists whenever a previously hidden provider is exposed for local testing. Prevention: for any debug-only or newly surfaced provider, audit both the metadata path and the bundle/copy pipeline before marking the slice done.
- 2026-03-22: Settings-driven provider reveals in the tray were still too fragile because one transient `tray:navigate` event was carrying both the selection change and the reload intent; if the main tray window missed that event timing, reopening the bar kept the old provider and stale data. Fix: persist the last explicit tray target view in Tauri, let the tray frontend consume/replay it on mount and focus, and pass the explicit target into the provider refresh path. Prevention: for cross-window navigation in Tauri, do not rely on one fire-and-forget event to carry state that must survive focus/show timing; keep a small backend replay buffer for the last explicit target.
- 2026-03-22: The standalone Settings window reused tray-sized segmented controls and a tab header with a hard minimum-width action area, so narrower widths crushed labels and made provider rows over-truncate even though the page technically fit. Fix: let the header stack earlier, move settings controls to responsive grid layouts, and allow provider subtitles/badges to wrap instead of forcing single-line squeeze. Prevention: for desktop settings pages, do not keep tray-style horizontal control groups once the surface can be resized or embedded at narrower widths; use responsive grid breakpoints and wrapping text by default.
- 2026-03-22: The tray popup still had first-open and overflow-menu geometry drift because the frontend owned the measured target height while Rust only remembered the tray anchor; menu-based opens also reused stale `show_panel` behavior that never repositioned from the current cursor context. Fix: store the last measured logical panel height in Rust via `sync_panel_geometry`, collapse the frontend resize flow to one settled update per height, and route tray menu opens through a cursor-based positioning helper. Prevention: for tray apps, keep native window geometry memory in one backend-owned place and make every non-tray-rect open path choose an explicit fallback anchor instead of assuming a previous tray click exists.
- 2026-03-22: The extracted standalone settings window left one stale tray-menu path behind: `Go to Settings` still tried to navigate the tray popup to a removed settings route. Fix: route tray-menu settings opens directly to `open_settings_window` and keep the tray popup usage-only. Prevention: when moving a workflow into a separate window, audit every backend menu/event entry point as well as the frontend buttons, or an old route can survive outside the visible React navigation.
- 2026-03-22: Providers that rely on startup probing can still feel view-coupled if the app only retries missing data for the currently active detail view on focus. Fix: add a generic enabled-provider catch-up batch for eligible providers that appear without probe state, so background initialization stays independent of which detail page is open. Prevention: keep provider startup/loading rules keyed off enabled/supported state, not off current view selection.
- 2026-03-21: `AGENTS.md` had drifted into a mixed template/local state with mojibake, stale command guidance, and one blank documentation path, which made the repo playbook less trustworthy during execution. Fix: rewrite the file against the live repo (`README.md`, `package.json`, current docs workflow) and keep only instructions that match the actual UsageBar toolchain and operating rules. Prevention: when editing shared agent instructions, validate each rule against current repo reality and remove any line that references unavailable tools, broken placeholders, or unclear ownership.
- 2026-03-21: Moving this repo to `D:\UsageBar\usagebar` while keeping the copied `src-tauri/target` directory left Cargo/Tauri `root-output` metadata pointing at `D:\UsageBar\openusage`, so `tauri dev` failed in the custom build step while reading autogenerated permissions from a path that no longer existed. Fix: teach the Windows `tauri` wrapper to detect moved-repo `target/debug/build/*/root-output` entries and clear only regenerable debug build metadata before launching Tauri; cover it with a focused Node regression test. Prevention: when a Rust/Tauri repo is renamed or copied with `target/` intact, treat absolute-path `root-output` files as stale cache inputs and auto-prune them before trusting cached build metadata.
- 2026-03-18: The standalone Settings provider list reused the tray-selection callback shape and passed `revealInTray: true` on ordinary row clicks, so choosing a provider to edit unexpectedly switched/opened the main bar view; the selected card also left the checkbox too low-contrast against the light selection fill. Fix: keep provider selection in Settings as a local id-only state change and give the selected-row checkbox its own contrasting border/fill overrides. ## 2026-03-10

- Issue: provider setup existed both in `Settings` and in the main provider detail view, splitting the setup UX across two surfaces.
  Fix: removed the settings copy and kept provider setup only on the provider detail page.
  Prevention: when a workflow already lives on the provider's primary surface, avoid mirroring the full control path in settings; settings should keep global preferences only.
- Issue: provider setup could mount expanded during the initial probe-loading phase, then collapse as soon as a successful provider state arrived, which caused a visible tray-panel height jolt on Windows.
  Fix: start the setup panel collapsed while probe loading is in progress, then only auto-expand after loading settles still-disconnected.
  Prevention: native tray-panel content that affects window height must not auto-expand from provisional loading state; wait for settled data before applying first-render expansion.
- Issue: provider-secret metadata can say a secret exists even when the underlying Windows credential entry is already missing, which made `Clear secret` fail silently and block cleanup.
  Fix: tolerate Windows missing-entry delete variants in Tauri, log save/delete failures explicitly, and keep the UI surfacing the backend error text.
  Prevention: for credential-store deletes, treat platform-specific "not found" errors as idempotent success and always log native failures at the frontend/backend boundary.
- Issue: even after native resize anchoring was stabilized, the panel could still look like it was falling downward because the inner shell animated its own height and sparse states let the nav column collapse too far.
  Fix: remove the shell-level CSS height transition and enforce a nav-based minimum logical panel height derived from `Home + enabled providers + Help + Settings`.
  Prevention: tray-panel resize polish must treat native window movement and inner layout animation separately; never animate both layers at once, and keep a minimum height for always-visible navigation.
- Issue: upward panel growth could still show a brief downward jump because the app resized the native window before moving it to the new anchored position.
  Fix: when the target height grows, reposition first using the final height and resize second; keep resize-first for non-growing updates.
  Prevention: for bottom-anchored tray panels, growth updates should use `move -> resize` so the window reads as rising from the tray instead of dropping then correcting.
