# Lessons

- 2026-03-18: The standalone Settings provider list reused the tray-selection callback shape and passed `revealInTray: true` on ordinary row clicks, so choosing a provider to edit unexpectedly switched/opened the main bar view; the selected card also left the checkbox too low-contrast against the light selection fill. Fix: keep provider selection in Settings as a local id-only state change and give the selected-row checkbox its own contrasting border/fill overrides. Prevention: when a settings surface mirrors tray navigation, do not reuse navigation-only side effects by default, and visually test embedded controls against both selected and unselected row fills.
- 2026-03-18: Antigravity on Windows can expose multiple localhost ports with mixed protocols at once (`https` LS, `http` LS, dead sockets, plus an extension port that may return `404`), so treating the first non-throwing probe as authoritative made provider status intermittently wrong even after the localized `netstat` parser was fixed. Fix: iterate through the discovered port/scheme candidates until one returns real usage data, and cover the mixed-port case with a focused plugin test. Prevention: for local daemon discovery, do not commit to a candidate based on a shallow health probe alone; require the actual business endpoint to return usable data before stopping the search.
- 2026-03-18: The standalone settings-window flow made it easier to leave a hidden local dev `openusage.exe` running in the tray, and a later `bun run tauri dev` then failed immediately with `failed to remove ... openusage.exe` / `os error 5` before Cargo could rebuild. Fix: route the local Tauri package script through a Windows-aware wrapper that kills only stale workspace-local debug binaries before `tauri dev`. Prevention: for tray apps with hidden-window flows, guard the dev launcher against exact-path stale binaries instead of assuming the previous debug child always exited with the shell.
- 2026-03-17: After moving Windows provider-secret persistence off Credential Manager, the settings UI still reported `system credential vault` on successful saves, which pointed debugging at the wrong backend. Fix: switch the success copy to backend-agnostic wording and record the DPAPI local-store choice in repo notes. Prevention: when secret-storage backend changes, audit both success and failure copy so the UI never describes a retired storage mechanism.
- 2026-03-17: OpenCode setup text was too vague: users could see both `OpenCode` and `OpenCode Go` without any repo-level explanation of why both exist, and "copy the cookie from billing" did not tell them to capture the full request `Cookie` header from the authenticated billing or `/_server` request. Fix: add a dedicated OpenCode provider doc, cross-link it with OpenCode Go, and make the settings copy explicit about the website flow and request-header capture steps. Prevention: when a provider depends on an internal authenticated web flow, document the exact page, exact request header name, and why similarly named providers are separate products/sources.
- 2026-03-17: OpenCode's old `usage data not found for this workspace` message collapsed two very different cases: `null` subscription data vs a non-null billing payload that no longer matched the expected `rollingUsage`/`weeklyUsage` shape. Fix: keep the `null` case separate and make the parse-failure case name the missing fields and point the user at verifying the `wrk_...` workspace ID before assuming auth is broken. Prevention: for internal web payload scrapes, report the missing fields and the likely operator action instead of one generic parse error.
- 2026-03-17: Windows provider-secret storage was harder to debug than necessary because `keyring::Entry::new(service, user)` does not give Windows a clean namespaced vault entry; the backend flattens it into a single target string (`username.service`), which made the OpenUsage provider-secret target opaque and likely contributed to the save/read confusion. Fix: use `Entry::new_with_target(...)` for Windows provider secrets so the Credential Manager target name is explicit, and keep the old implicit mapping as a read/delete fallback. Prevention: when storing app-owned secrets in Windows Credential Manager, prefer an explicit target name over `keyring`'s implicit Windows mapping so the entry is predictable, inspectable, and stable across code paths.
- 2026-03-17: Windsurf on this Windows machine already had valid signed-in state in `%AppData%\\Windsurf\\User\\globalStorage\\state.vscdb`, but the plugin still looked only at macOS SQLite and app-bundle paths, so it could never reach either the local LS metadata path or the cloud fallback correctly. Fix: make the plugin choose Windows state DB paths, Windows LS process/OS metadata, and Windows `product.json` version fields (`windsurfVersion` + `codeiumVersion`) before declaring Windsurf unavailable. Prevention: for VS Code-derived providers, audit the Windows app-data path, install metadata file, and LS process name together; fixing only one of those surfaces still leaves the provider effectively blocked.
- 2026-03-17: Precise provider-secret save diagnostics could still collapse to the useless fallback `Failed to save secret.` because the settings UI only trusted `Error` objects, while Tauri command failures can arrive as plain strings. Fix: normalize unknown/string-shaped errors before rendering save/delete/config messages, and cover the path with a focused settings test. Prevention: when surfacing Tauri command failures in React, extract messages from `string`, `Error`, and `{ message }` shapes instead of assuming one runtime error type.
- 2026-03-17: Claude on this Windows machine no longer had the legacy `~/.claude/.credentials.json` OAuth file, but it still kept real signed-in account state in `~/.claude.json`; treating the missing legacy file as logged-out would have been a false negative. Fix: read `~/.claude.json` as a signed-in fallback source, use its account metadata to avoid false `Not logged in`, and keep usage sourcing restricted to OAuth tokens or local ccusage. Prevention: when a provider migrates local auth storage, separate "signed-in account present" from "has the exact token needed for remote usage APIs" instead of collapsing both into one credential-file check.
- 2026-03-17: Ollama save could still look successful right before an immediate retry reported `provider secret not found` because the Tauri write path verified the secret with the same keyring entry object it had just written, while the plugin probe later reopened the credential through a fresh entry lookup. Fix: reopen the provider-secret entry for read-after-write verification and cover the fresh-lookup helper with focused Rust tests. Prevention: when validating OS-vault writes, verify through the same fresh lookup path the runtime reader will use; same-handle read-back is not a strong enough persistence check.
- 2026-03-17: Copilot could still say `Not logged in` on Windows even with a valid `gh` session because GitHub CLI's own auth command worked while OpenUsage's narrower credential-read path missed the usable token source on that machine. Fix: keep the keychain-based paths first, but fall back to `gh auth token` through the host API when those reads miss, and cover it with a focused Copilot regression test. Prevention: when integrating with another CLI's authentication, verify both the external credential store path and the CLI's own authoritative token command before concluding the user is signed out.
- 2026-03-17: Antigravity on this Windows box fell back to stale Cloud Code quota data because the host LS port parser only accepted the English `LISTENING` token from `netstat`, while the localized system returned `ABHÖREN`; the extension port then produced the misleading `100% left` path instead of the real local LS numbers. Fix: detect Windows listening rows by PID plus the listen-style foreign address (`0.0.0.0:0` / `[::]:0`) instead of depending on one localized state string, and cover it with a Rust regression test. Prevention: never key Windows process/network parsing to English-only shell output when the same structural signal is available in locale-stable columns.
- 2026-03-17: The tray popup could keep a too-small hidden-startup height until a later interaction changed the content enough to trigger another resize. Fix: trigger a panel re-measure when the popup window gains focus and lock it in with a frontend regression test. Prevention: when a Tauri tray window is measured while hidden, re-run the size calculation on first focus/show instead of assuming the hidden layout was final.

- 2026-03-17: Amp stayed blocked on Windows longer than necessary because the plugin path looked Linux-specific at a glance, even though Amp's own Windows docs use the same home-relative `.local/share/amp/secrets.json` location. Fix: keep the existing path, add a Windows-specific regression test, and move the provider to experimental instead of inventing a second Windows path. Prevention: before branching on OS-specific config paths, check whether the official tool already documents the same home-relative location on Windows.
- 2026-03-17: Copilot in this Windows fork could never reliably follow the active `gh` account because the plugin host keychain wrapper only exposed OpenUsage-scoped entries, while GitHub CLI stores per-account credentials under external service/account pairs. Fix: add an account-aware external credential read, parse the active login from `hosts.yml`, and tag the Copilot cache with that login so stale cached tokens are ignored after `gh auth switch`. Prevention: when a provider depends on another tool's credential store, verify the host API can address that tool's exact service/account shape before assuming a service-name lookup is sufficient.
- 2026-03-17: Gemini on Windows already had the main OAuth credential files under `~/.gemini`, but expired-session refresh still failed because the plugin only searched Unix-oriented `oauth2.js` install paths. Fix: add the common Windows npm-global `~/AppData/Roaming/npm/node_modules/...` candidates and cover the refresh path with a focused test. Prevention: when a provider reuses CLI OAuth files across platforms, audit both the token file path and the CLI-install path used to recover client credentials before calling Windows support complete.

- 2026-03-16: Moving provider management into a standalone settings window broke the old implicit tray-panel coupling; provider selection updated only local settings state, so the main bar never reopened on that provider. Fix: add an explicit settings-to-tray bridge for user-triggered provider selection while keeping passive default selection local-only. Prevention: when extracting a workflow into a separate window, inventory every cross-window side effect the old surface provided and reintroduce the ones users still expect with explicit commands/events.

- 2026-03-16: Reusing the tray popup for settings kept dragging desktop-preferences UX constraints back into a surface that should stay focused on quick usage checks. Fix: move all settings entry points to a dedicated settings window and keep the tray popup usage-only. Prevention: when a flow needs deeper configuration, scrolling, and provider management, make it a separate window early instead of stretching the tray panel beyond its job.
- 2026-03-16: A separate settings window still felt wrong when it behaved like a fixed-height dashboard with nested scroll regions. Fix: let the settings window expand as a normal document and make the whole page scroll instead of locking the shell height and scrolling inner panes. Prevention: for desktop preferences pages, default to page-level scrolling unless a pane truly needs independent scrolling for a deliberate workflow reason.

- 2026-03-11: Rust test runs that redirect `CARGO_TARGET_DIR` into ad-hoc folders like `src-tauri/target-test-temp*` can flood `git status` with thousands of fake changes. Fix: ignore the temporary target-folder pattern next to the normal Cargo `target/` rule. Prevention: whenever tests/build scripts use alternate Cargo target dirs inside the repo, add a scoped ignore rule immediately.

- 2026-03-10: Panel-Repositioning darf bei laufendem Resize nicht auf eine frisch aus dem OS zurückgelesene Fensterhöhe oder eine jedes Mal neu rekonstruierte Tray-Unterkante vertrauen. Fix: Zielhöhe aus dem Frontend an `reposition_panel` übergeben, einen expliziten vertikalen Panel-Anker speichern und für Taskleisten-Clamps `workArea` statt voller Monitorgröße verwenden. Prevention: Bei anchor-basierten Desktop-Panels die feste Kante explizit speichern und Resize-/Reposition-Parameter aus derselben Quelle berechnen.
- 2026-03-10: Native Per-Frame-Resize-Animationen eines Tray-Panels erzeugen sichtbares Jitter selbst mit korrektem Anchor. Fix: Fensterhöhe nur noch einmal pro final gemessener Zielhöhe setzen und danach einmal repositionieren. Prevention: Native Fensterbewegungen bei Menu-Bar/Tray-Panels nicht mit `requestAnimationFrame` animieren, außer die Plattform garantiert atomische Bounds-Updates.

## 2026-03-10

- Issue: provider setup existed both in `Settings` and in the main provider detail view, splitting the setup UX across two surfaces.
  Fix: removed the settings copy and kept provider setup only on the provider detail page.
  Prevention: when a workflow already lives on the provider's primary surface, avoid mirroring the full control path in settings; settings should keep global preferences only.
- Issue: provider setup could mount expanded during the initial probe-loading phase, then collapse as soon as a successful provider state arrived, which caused a visible tray-panel height jolt on Windows.
  Fix: start the setup panel collapsed while probe loading is in progress, then only auto-expand after loading settles still-disconnected.
  Prevention: native tray-panel content that affects window height must not auto-expand from provisional loading state; wait for settled data before applying first-render expansion.
- Issue: provider-secret metadata can say a secret exists even when the underlying Windows credential entry is already missing, which made `Clear secret` fail silently and block cleanup.
  Fix: tolerate Windows missing-entry delete variants in Tauri, log save/delete failures explicitly, and keep the UI surfacing the backend error text.
  Prevention: for credential-store deletes, treat platform-specific "not found" errors as idempotent success and always log native failures at the frontend/backend boundary.
- Issue: even after native resize anchoring was stabilized, the panel could still look like it was falling downward because the inner shell animated its own height and sparse states let the nav column collapse too far.
  Fix: remove the shell-level CSS height transition and enforce a nav-based minimum logical panel height derived from `Home + enabled providers + Help + Settings`.
  Prevention: tray-panel resize polish must treat native window movement and inner layout animation separately; never animate both layers at once, and keep a minimum height for always-visible navigation.
- Issue: upward panel growth could still show a brief downward jump because the app resized the native window before moving it to the new anchored position.
  Fix: when the target height grows, reposition first using the final height and resize second; keep resize-first for non-growing updates.
  Prevention: for bottom-anchored tray panels, growth updates should use `move -> resize` so the window reads as rising from the tray instead of dropping then correcting.
- Issue: direct one-step growth fixed the anchor but still felt abrupt, while smoothing shrink paths could violate tight work-area clamps.
  Fix: add a very short 2-3 step tween for growth only; keep shrink/clamped reductions immediate.
  Prevention: on Windows tray panels, smooth only the upward-growth path and never interpolate through heights that would exceed the current monitor cap.
- Issue: the Windows credential-delete tolerance test passed mixed-case native error strings, but the matcher only worked when callers lowercased the message first, which made the helper brittle and broke the Rust suite.
  Fix: normalize case inside `is_missing_credential_error` before checking known platform variants.
  Prevention: helper functions that classify native error text should normalize their own input instead of relying on every caller and test to pre-process strings.
- Issue: Windows keyring delete can also return `No matching entry found in secure storage`, which is still an idempotent missing-entry case; the matcher treated it as a hard failure and blocked UI cleanup of stale secret metadata.
  Fix: extend `is_missing_credential_error` to accept the `no matching entry found` variant and cover it with a Rust test.
  Prevention: when treating native credential deletes as idempotent, include semantically equivalent provider/backend wording variants in regression tests, not just one platform string.
- Issue: Ollama setup could report `Secret stored in the system credential vault` while a later probe still collapsed to the generic paste-cookie prompt, hiding whether the vault entry was missing or unreadable.
  Fix: verify provider-secret writes with an immediate read-back in Tauri, and make the Ollama plugin surface concrete credential-vault read errors instead of always rethrowing the generic setup message.
  Prevention: for credential-backed provider setup, success toasts must only happen after a read-back check, and plugin secret-read failures should preserve the real backend error category in user-visible probe output.
- Issue: the plugin-host `providerSecrets.read()` helper swallowed every keyring `get_password()` failure and downgraded it to `provider secret not found`, so the UI could not distinguish a missing vault entry from a real Windows credential read error.
  Fix: classify only known missing-entry variants as `not found` and bubble all other keyring read errors through the host API unchanged.
  Prevention: secret-read adapters must not collapse all backend failures into a single not-found state; preserve non-idempotent errors so provider setup can show actionable diagnostics.
